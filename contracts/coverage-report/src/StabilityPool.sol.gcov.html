      18                 :            : /*
      19                 :            :  * The Stability Pool holds Bold tokens deposited by Stability Pool depositors.
      20                 :            :  *
      21                 :            :  * When a trove is liquidated, then depending on system conditions, some of its Bold debt gets offset with
      22                 :            :  * Bold in the Stability Pool:  that is, the offset debt evaporates, and an equal amount of Bold tokens in the Stability Pool is burned.
      23                 :            :  *
      24                 :            :  * Thus, a liquidation causes each depositor to receive a Bold loss, in proportion to their deposit as a share of total deposits.
      25                 :            :  * They also receive an Coll gain, as the collateral of the liquidated trove is distributed among Stability depositors,
      26                 :            :  * in the same proportion.
      27                 :            :  *
      28                 :            :  * When a liquidation occurs, it depletes every deposit by the same fraction: for example, a liquidation that depletes 40%
      29                 :            :  * of the total Bold in the Stability Pool, depletes 40% of each deposit.
      30                 :            :  *
      31                 :            :  * A deposit that has experienced a series of liquidations is termed a "compounded deposit": each liquidation depletes the deposit,
      32                 :            :  * multiplying it by some factor in range ]0,1[
      33                 :            :  *
      34                 :            :  *
      35                 :            :  * --- IMPLEMENTATION ---
      36                 :            :  *
      37                 :            :  * We use a highly scalable method of tracking deposits and Coll gains that has O(1) complexity.
      38                 :            :  *
      39                 :            :  * When a liquidation occurs, rather than updating each depositor's deposit and Coll gain, we simply update two state variables:
      40                 :            :  * a product P, and a sum S.
      41                 :            :  *
      42                 :            :  * A mathematical manipulation allows us to factor out the initial deposit, and accurately track all depositors' compounded deposits
      43                 :            :  * and accumulated Coll gains over time, as liquidations occur, using just these two variables P and S. When depositors join the
      44                 :            :  * Stability Pool, they get a snapshot of the latest P and S: P_t and S_t, respectively.
      45                 :            :  *
      46                 :            :  * The formula for a depositor's accumulated Coll gain is derived here:
      47                 :            :  * https://github.com/liquity/dev/blob/main/papers/Scalable_Reward_Distribution_with_Compounding_Stakes.pdf
      48                 :            :  *
      49                 :            :  * For a given deposit d_t, the ratio P/P_t tells us the factor by which a deposit has decreased since it joined the Stability Pool,
      50                 :            :  * and the term d_t * (S - S_t)/P_t gives us the deposit's total accumulated Coll gain.
      51                 :            :  *
      52                 :            :  * Each liquidation updates the product P and sum S. After a series of liquidations, a compounded deposit and corresponding Coll gain
      53                 :            :  * can be calculated using the initial deposit, the depositor’s snapshots of P and S, and the latest values of P and S.
      54                 :            :  *
      55                 :            :  * Any time a depositor updates their deposit (withdrawal, top-up) their accumulated Coll gain is paid out, their new deposit is recorded
      56                 :            :  * (based on their latest compounded deposit and modified by the withdrawal/top-up), and they receive new snapshots of the latest P and S.
      57                 :            :  * Essentially, they make a fresh deposit that overwrites the old one.
      58                 :            :  *
      59                 :            :  *
      60                 :            :  * --- SCALE FACTOR ---
      61                 :            :  *
      62                 :            :  * Since P is a running product in range ]0,1] that is always-decreasing, it should never reach 0 when multiplied by a number in range ]0,1[.
      63                 :            :  * Unfortunately, Solidity floor division always reaches 0, sooner or later.
      64                 :            :  *
      65                 :            :  * A series of liquidations that nearly empty the Pool (and thus each multiply P by a very small number in range ]0,1[ ) may push P
      66                 :            :  * to its 18 digit decimal limit, and round it to 0, when in fact the Pool hasn't been emptied: this would break deposit tracking.
      67                 :            :  *
      68                 :            :  * So, to track P accurately, we use a scale factor: if a liquidation would cause P to decrease to <1e-9 (and be rounded to 0 by Solidity),
      69                 :            :  * we first multiply P by 1e9, and increment a currentScale factor by 1.
      70                 :            :  *
      71                 :            :  * The added benefit of using 1e9 for the scale factor (rather than 1e18) is that it ensures negligible precision loss close to the
      72                 :            :  * scale boundary: when P is at its minimum value of 1e9, the relative precision loss in P due to floor division is only on the
      73                 :            :  * order of 1e-9.
      74                 :            :  *
      75                 :            :  * --- EPOCHS ---
      76                 :            :  *
      77                 :            :  * Whenever a liquidation fully empties the Stability Pool, all deposits should become 0. However, setting P to 0 would make P be 0
      78                 :            :  * forever, and break all future reward calculations.
      79                 :            :  *
      80                 :            :  * So, every time the Stability Pool is emptied by a liquidation, we reset P = 1 and currentScale = 0, and increment the currentEpoch by 1.
      81                 :            :  *
      82                 :            :  * --- TRACKING DEPOSIT OVER SCALE CHANGES AND EPOCHS ---
      83                 :            :  *
      84                 :            :  * When a deposit is made, it gets snapshots of the currentEpoch and the currentScale.
      85                 :            :  *
      86                 :            :  * When calculating a compounded deposit, we compare the current epoch to the deposit's epoch snapshot. If the current epoch is newer,
      87                 :            :  * then the deposit was present during a pool-emptying liquidation, and necessarily has been depleted to 0.
      88                 :            :  *
      89                 :            :  * Otherwise, we then compare the current scale to the deposit's scale snapshot. If they're equal, the compounded deposit is given by d_t * P/P_t.
      90                 :            :  * If it spans one scale change, it is given by d_t * P/(P_t * 1e9). If it spans more than one scale change, we define the compounded deposit
      91                 :            :  * as 0, since it is now less than 1e-9'th of its initial value (e.g. a deposit of 1 billion Bold has depleted to < 1 Bold).
      92                 :            :  *
      93                 :            :  *
      94                 :            :  *  --- TRACKING DEPOSITOR'S Coll GAIN OVER SCALE CHANGES AND EPOCHS ---
      95                 :            :  *
      96                 :            :  * In the current epoch, the latest value of S is stored upon each scale change, and the mapping (scale -> S) is stored for each epoch.
      97                 :            :  *
      98                 :            :  * This allows us to calculate a deposit's accumulated Coll gain, during the epoch in which the deposit was non-zero and earned Coll.
      99                 :            :  *
     100                 :            :  * We calculate the depositor's accumulated Coll gain for the scale at which they made the deposit, using the Coll gain formula:
     101                 :            :  * e_1 = d_t * (S - S_t) / P_t
     102                 :            :  *
     103                 :            :  * and also for scale after, taking care to divide the latter by a factor of 1e9:
     104                 :            :  * e_2 = d_t * S / (P_t * 1e9)
     105                 :            :  *
     106                 :            :  * The gain in the second scale will be full, as the starting point was in the previous scale, thus no need to subtract anything.
     107                 :            :  * The deposit therefore was present for reward events from the beginning of that second scale.
     108                 :            :  *
     109                 :            :  *        S_i-S_t + S_{i+1}
     110                 :            :  *      .<--------.------------>
     111                 :            :  *      .         .
     112                 :            :  *      . S_i     .   S_{i+1}
     113                 :            :  *   <--.-------->.<----------->
     114                 :            :  *   S_t.         .
     115                 :            :  *   <->.         .
     116                 :            :  *      t         .
     117                 :            :  *  |---+---------|-------------|-----...
     118                 :            :  *         i            i+1
     119                 :            :  *
     120                 :            :  * The sum of (e_1 + e_2) captures the depositor's total accumulated Coll gain, handling the case where their
     121                 :            :  * deposit spanned one scale change. We only care about gains across one scale change, since the compounded
     122                 :            :  * deposit is defined as being 0 once it has spanned more than one scale change.
     123                 :            :  *
     124                 :            :  *
     125                 :            :  * --- UPDATING P WHEN A LIQUIDATION OCCURS ---
